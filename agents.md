# agents.md

## Назначение

Этот файл задаёт правила работы для CODEX/LLM-агентов в проекте `promotion_Renata`.

Агент обязан сначала читать документацию из папки `docs/`, а уже потом предлагать изменения в коде.

Если документация и код расходятся, агент должен:
1. считать код фактическим runtime-источником истины;
2. отметить расхождение в отчёте;
3. при необходимости предложить минимальное обновление документации.

---

## Что читать в первую очередь

Перед любыми изменениями агент должен последовательно прочитать:

1. `docs/architecture.md`
2. `docs/principles.md`
3. `docs/codex-agents.md`

После этого агент изучает нужные директории проекта:

- backend: `core/`
- bot: `telegram_bot/`
- crm frontend: `crm_web/admin-panel/`
- integrations: `integrations/`, `core/api/webhooks.py`
- rag/ai: `core/rag/`, `core/ai/`, `rag_data/`
- infra: `compose.prod.yml`, Dockerfile, nginx-конфиги, `scripts/`

---

## Краткий контекст проекта

Проект: CRM + Telegram Bot + AI/RAG для психолога Ренаты.

Текущая архитектура проекта:
- backend на FastAPI;
- Telegram bot на `python-telegram-bot`;
- CRM frontend на React/Vite;
- PostgreSQL вынесен вне compose;
- прод-развёртывание через Docker Compose;
- GetCourse работает в режиме `webhook-only`;
- AI работает через OpenRouter + локальный RAG;
- базовый архитектурный принцип: `core-first`, где `core/` — единый источник бизнес-логики.

---

## Главные правила для любого агента

1. Делать минимальные правки.
2. Не ломать API-контракты, docker compose, существующие entrypoint’ы и env-схему без крайней необходимости.
3. Не логировать секреты: `OPENROUTER_API_KEY`, `BOT_TOKEN`, `JWT_SECRET`, `DATABASE_URL`, webhook tokens, invite links, PII.
4. Все исходники и новые файлы — только в UTF-8.
5. Не переносить бизнес-логику во frontend или bot, если она должна жить в `core/`.
6. Для интеграций использовать явные boundary: внешние системы -> API/webhook -> `core/services`.
7. Любую спорную правку сопровождать коротким отчётом:
   - что изменено;
   - какие файлы затронуты;
   - как проверить;
   - какие риски остались.

## Разделение CODEX на агентов

Полное описание ролей лежит в `docs/codex-agents.md`.

По умолчанию использовать такую специализацию:

- **Agent 1 — Architecture Guard**  
  Проверяет, что направление зависимостей не нарушено и логика остаётся в `core/`.

- **Agent 2 — Backend/API Agent**  
  Работает с FastAPI, SQLAlchemy, webhook’ами, auth, CRM API.

- **Agent 3 — Telegram Bot Agent**  
  Работает с `telegram_bot/`, UX ветками, assistant mode, платежными/меню-сценариями.

- **Agent 4 — CRM Frontend Agent**  
  Работает с `crm_web/admin-panel/`, таблицами, фильтрами, auth UI, аналитикой.

- **Agent 5 — AI/RAG Agent**  
  Работает с OpenRouter, retrieval pipeline, `core/rag/`, `rag_data/`, AI diagnostics.

- **Agent 6 — Infra/Deploy Agent**  
  Работает с `compose.prod.yml`, nginx, healthchecks, deploy/runbook, scripts.

- **Agent 7 — QA/Smoke Agent**  
  Не вносит архитектурных изменений, а проверяет сборку, smoke, health, сценарии регрессии.

## Обязательный формат ответа агента после работы

Каждый агент должен возвращать итог в таком формате:

### 1. Что сделано
Кратко и по делу.

### 2. Какие файлы изменены
Список путей.

### 3. Почему изменения корректны
1–5 пунктов по архитектуре и runtime.

### 4. Как проверить
Конкретные команды.

### 5. Риски / что не покрыто
Только реальные риски.

## Стоп-условия

Агент должен остановиться и явно предупредить, если задача требует:
- массового рефакторинга нескольких слоёв без необходимости;
- изменения секретов, продовых ключей или приватных URL;
- удаления рабочих runtime-модулей без подтверждённого плана миграции;
- изменения схемы БД без обратимой или безопасной стратегии применения.
